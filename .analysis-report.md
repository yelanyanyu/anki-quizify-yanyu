# Anki NestedReveal 兼容性问题分析报告

## 一、问题摘要

在 Anki 25.09.2 (Qt 6.9.1, Chromium 122) 中，`QuizifyNestedReveal` 模块的核心交互——按钮点击——**完全失效**。相同的代码在独立浏览器（Chrome, test.html）中一切正常。按钮能渲染显示，但三种不同的点击绑定方式全部失败。

关键症状：
- 按钮 HTML 正确渲染（可见可读）
- 点击按钮无任何反应（无报错，无视觉反馈，无 console 输出）
- 不仅我们的按钮无效，连 `<button onclick="...;return false;">` 这种标准 HTML 写法也无效
- 原始模板中已存在的交互（填空、选择、折叠）的 onclick 正常——说明差异在于**动态创建 vs 静态 HTML 中的事件绑定方式**

## 三、点击事件：四种方案全部失败

### 方案 1：`innerHTML` + inline `onclick` 属性（带 `return false`）

```javascript
html += '<button onclick="QuizifyNestedReveal.handleUnknown();return false;">不认识</button>';
ctrl.innerHTML = html;
```

**结果**：按钮显示正常，点击无效。

### 方案 2：`btn.onclick = handlerFunction`（DOM API 创建按钮）

```javascript
var btn = document.createElement('button');
btn.textContent = '不认识';
btn.onclick = handleUnknown;
ctrl.appendChild(btn);
```

**结果**：按钮显示正常，点击无效（仅第二个按钮偶然生效，推测是巧合）。

### 方案 3：`btn.addEventListener('click', handlerFunction)`（DOM API）

```javascript
btn.addEventListener('click', handleUnknown);
```

**结果**：所有按钮连一个都不生效了。

### 方案 4：事件委托（在容器上监听 click，通过 `e.target.closest()` 分发）

```javascript
ctrl.addEventListener('click', function(e) {
    e.preventDefault();
    var btn = e.target.closest('button');
    if (btn.classList.contains('quizify-btn--unknown')) handleUnknown();
    ...
});
```

**结果**：第一次点击有效。但调用 `buildControls` 更新按钮（`innerHTML` 替换）后，所有按钮失效——包括重新渲染后的按钮。推测 `innerHTML` 替换在 Qt 中会影响父容器的事件监听器状态。

## 四、已发现并绕过的 Anki 兼容性问题

### 4.1 HTML 闭合标签在 JS 字符串中导致 script 块静默丢弃

**现象**：当 `<script>` 块内的 JS 字符串包含 `'</div>'`, `'</span>'`, `'</audio>'` 等 HTML 闭合标签时，**整个 script 块被 Anki 的模板渲染管线静默丢弃**——不报任何错误。

**二分定位证据**：

| Script 块内容 | 行数 | 结果 |
|---|---|---|
| `parseExamples` 函数 | ~10 行 | ✅ 执行 |
| + `buildItemHTML`（含 `</audio>`、`</span>`、`</div>` 字符串） | ~25 行 | ❌ 丢弃 |
| `buildItemHTML`（使用 DOM API，无闭合标签字符串） | ~25 行 | ✅ 执行 |
| 三者拆分到独立 `<script>` 块 | 各 ~10 行 | ✅ 全部执行 |

**根因**：Anki 用 `.innerHTML` 设卡片内容后，手动提取并重新注入 `<script>` 标签（见 [reviewer/index.ts](https://github.com/ankitects/anki/blob/main/ts/reviewer/index.ts) 和 [issue #1120](https://github.com/ankitects/anki/issues/1120)）。在此提取过程中，Qt WebEngine 的 HTML 解析器对 `</...>` 模式的处理有 bug。

**绕过**：用 `document.createElement` + `element.outerHTML` 替代字符串拼接；剩余的字符串用 `<\/` 转义。

### 4.2 字段名不支持连字符

`{{example-list}}` → 改为 `{{ExampleList}}`

### 4.3 `sessionStorage` 旧版 Anki 不可用

`sessionStorage.setItem()` → 改为 `window.__quizifyEase = value`

## 五、推测的未解决的兼容性问题

### 5.1 Content Security Policy 阻断动态 onclick（最有可能）

Anki 在 CVE-2024-32484 / CVE-2025-43703 之后加强了安全策略：
- Issue [#3935](https://github.com/ankitects/anki/issues/3935) — DOMPurify 清洗 + CSP 变更
- Anki 可能启用了 CSP 的 `script-src` 策略，禁止或限制 `unsafe-inline`
- 如果 CSP 限制了 inline event handlers，动态创建的 `onclick` 会被浏览器静默忽略
- **但原始模板的静态 onclick 正常**——说明要么 CSP 区分静态 HTML 和动态注入的 handlers，要么有白名单机制

### 5.2 Qt WebEngine 的 `innerHTML` 不激活 inline event handlers

Chromium 的正常行为是 `innerHTML` 会激活 inline event handlers。但 Qt WebEngine 可能有差异——如果 Anki 在注入 HTML 后做了额外的清洗步骤（如 DOMPurify），inline event handlers 可能被剥离。

### 5.3 `addEventListener` 在 Qt WebEngine 中不适用于动态元素

方案 3 和 4 的结果表明，Qt WebEngine 对 `addEventListener` 的支持可能不完整。这在 Qt 论坛（[ES6 modules silently ignored](https://forum.qt.io/topic/105537/es6-modules-in-qwebenginepage/3)）有类似报告——某些 JS API 在 Qt WebEngine 的 `setHtml()` 注入的页面中被静默忽略。

### 5.4 `<base>` 标签的副作用

Anki 设置 `<base href="...">` 指向媒体服务器（[commit 14f4107](https://github.com/ankitects/anki/commit/14f4107dfda3c75d8e0eeb1c44141ceb962cd3c6)），要求所有 onclick 必须 `return false` 防止页面导航。即使加了 `return false`，仍有可能与 `<base>` 有其他交互。

## 六、对比分析：原始模板的 onclick 为什么能工作

原始模板的 markup replacement 脚本（在 `{{^Skip Replace}}` 内）生成的交互元素——如：

```html
<span class="feedback" onclick="validateChoice(this)" data-correct-answers="AC">
```

这些 onclick 能正常工作。与我们动态按钮的关键差异：

| 维度 | 原始模板 | 我们的按钮 |
|------|---------|-----------|
| 注入方式 | `.innerHTML =`（在 `{{^Skip Replace}}` 脚本内） | `.innerHTML =`（在 IIFE 的 `buildControls` 内） |
| 注入时机 | 页面初次加载时 | 用户交互后动态更新 |
| 执行上下文 | 全局作用域 | IIFE 闭包内部 |
| onclick 目标 | 全局函数 `validateChoice` | IIFE 闭包内的 `handleUnknown` 等 |

最可能的原因：**原始模板的函数是全局函数（`function validateChoice(...)` 在 `<script>` 顶层定义），而 `QuizifyNestedReveal.handleUnknown()` 需要从 `onclick` 属性中通过 `window.QuizifyNestedReveal` 访问 IIFE 返回的对象**。

验证：如果 Anki 的 CSP 或其他安全机制阻止从 onclick 字符串中访问 IIFE 返回的对象，改为全局独立函数就能解决。

## 七、建议的修复方向（优先级排序）

### 方向 A：将 handler 暴露为全局函数（最可能有效）

把 `handleKnown`、`handleUnknown`、`handleEasy` 从 IIFE 内部移到全局作用域，通过 `window` 对象暴露，然后用 `onclick="handleUnknown();return false;"` 调用。

```html
<script>
// 全局独立 handler
function __nestedHandleUnknown() {
    window.QuizifyNestedReveal.handleUnknown();
}
</script>
```

验证方法：创建一个极简版，只测试一个 onclick 按钮能否调用全局函数。

### 方向 B：使用 `<a href="#">` + `onclick` 模式

原始模板使用的是 `onclick`，但 Closet 等 Anki 插件广泛使用 `<a>` 标签 + onclick 来规避按钮的兼容性问题。

### 方向 C：检查 Anki 的 CSP 设置

用 [AnkiWebView Inspector 插件](https://ankiweb.net/shared/info/31746032) 打开 Console，执行：
```javascript
console.log(document.querySelector('meta[http-equiv="Content-Security-Policy"]'));
```
查看是否有 CSP 限制。

### 方向 D：通过 Anki 的 `pycmd()` 桥接

Anki 提供 `pycmd('cmd_name')` 函数在前端和 Python 后端之间通信。一些模板使用 `pycmd` + Python addon 来实现按钮交互。虽然耦合度高，但如果其他方向都失败，这是最后手段。
